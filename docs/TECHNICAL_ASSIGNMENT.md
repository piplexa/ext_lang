# Техническое задание v2 (с нуля)

## Контекст

Документ основан на целях из `for_copilot_2.md`, но с новым правилом:
- функции и реализация из `initdb/02-ext-lang.sql` считаются несуществующими для новой версии;
- файл `initdb/02-ext-lang.sql` хранится как архив/backup для будущего изучения;
- новую систему нужно проектировать и реализовывать независимо от этого файла.

## Цель проекта

Сделать полноценное и производительное расширение PostgreSQL для мультиязычных значений с максимально прозрачным SQL-использованием:
- без явных вызовов служебных функций в обычных `SELECT`;
- без явных вызовов служебных функций в обычных `INSERT`;
- с явным обновлением переводов через одну короткую функцию в `UPDATE`.

## Платформа

- PostgreSQL 16+ (актуальные версии).
- Поддержка PostgreSQL 9.x и более ранних версий не требуется.

## Архитектурные требования

1. Базовый подход
- Можно использовать `hstore` как внутреннее хранилище переводов.
- Можно использовать C-расширение и собственный тип/обвязку ввода-вывода.
- Допускается SQL-слой только в объеме, необходимом для установки и служебных операций.

2. Прозрачность SQL
- Пользователь должен писать обычные запросы:
  - `SELECT column1 FROM table1;`
  - `INSERT INTO table1 (column1) VALUES ('привет');`
  - `UPDATE table1 SET column1 = ext_lang.u(column1, 'hello') WHERE id = 1;`
- Для `SELECT` и `INSERT` явные вызовы внутренних функций не должны требоваться.
- Для `UPDATE` языковых колонок допускается единая функция обновления без триггеров.

3. Приоритет производительности
- Главный приоритет: скорость `SELECT`.
- Допускается, что `INSERT` и `UPDATE` могут быть медленнее, если это улучшает прозрачность и скорость чтения.
- Решения должны минимизировать накладные расходы на путь чтения.

4. Хранение и fallback
- Значение должно поддерживать несколько языков в одном поле.
- Должен быть механизм выбора текущего языка сессии.
- Должен быть fallback (например: текущий язык -> язык по умолчанию -> служебный default).

## Ограничения

- `initdb/02-ext-lang.sql` нельзя изменять и нельзя использовать как зависимость новой логики.
- В новой реализации запрещено опираться на функции из `initdb/02-ext-lang.sql`.
- Допускается только чтение `initdb/02-ext-lang.sql` как исторического материала.

## Обязательные артефакты новой версии

1. Новый init-скрипт (или набор скриптов) для полностью независимой установки.
2. Расширение с собственными объектами (тип, функции, DDL), не конфликтующее с архивным SQL.
3. Документация по использованию прозрачных `SELECT/INSERT/UPDATE`.
4. Минимальный benchmark-сценарий для сравнения производительности чтения.

## Критерии приемки

1. Нулевые зависимости на функции из `initdb/02-ext-lang.sql`.
2. `SELECT/INSERT` работают в прозрачном режиме без явных служебных вызовов.
3. `UPDATE` языковой колонки выполняется через `ext_lang.u(...)` с сохранением существующих переводов.
4. `SELECT` демонстрирует приемлемую производительность на целевом объеме данных.
5. Все скрипты установки выполняются на чистом volume через Docker.
6. Архивный файл `initdb/02-ext-lang.sql` остается без изменений.

## Нефункциональные требования

- Предсказуемость поведения при смене языка сессии.
- Стабильность на конкурентных запросах.
- Четкая миграционная стратегия между версиями схемы расширения.
- Явные диагностические сообщения об ошибках конфигурации/языков.
